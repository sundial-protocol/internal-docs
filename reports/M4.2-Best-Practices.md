# Security Best Practices - Staking & Lending

## How the Repositories Work Together

`btc-locker` is a TypeScript library that constructs and validates Bitcoin transactions — it defines the on-chain rules: what scripts are valid, what keys are required, and what spending conditions must be met. `btc-yield` is the off-chain system (API + indexer) that orchestrates the yield protocol on top of those rules. The indexer watches the Bitcoin chain for transactions matching `btc-locker`'s Tapscript patterns and validates them against the protocol state stored in its database; the API exposes endpoints for providers and users to submit PSBTs and manage their positions. Neither system is independently sufficient — `btc-locker` enforces what is cryptographically possible, and `btc-yield` enforces the protocol rules on top of that. An attacker who bypasses the API still cannot spend funds without satisfying the on-chain script conditions defined by `btc-locker`, and an attacker who constructs a valid script spend still cannot manipulate the protocol state without also bypassing `btc-yield`'s indexer validation.

## btc-locker

- **Protocol-level security via Bitcoin Script:** Funds are locked in P2WSH outputs encoding two spend paths — an absolute timelock (`OP_CHECKLOCKTIMEVERIFY`) and a relative timelock (`OP_CHECKSEQUENCEVERIFY`) for depositor recovery, and an `OP_IF`/`OP_ELSE` escrow script requiring the counterparty key before the deadline. Neither party can spend unilaterally outside their designated path.
- **Key management & cryptography:** `KeyPair` and `keys.ts` enforce secp256k1 validity and PSBT-based signing so private keys never touch transaction construction directly.
- **Input validation:** `validation.ts` covers network-aware address checksum/format checks, pubkey hex, satoshi bounds, and locktime ranges — all with dedicated test coverage.
- **Trust model:** Formal threat assumptions and per-party capability boundaries are documented in `spec/v0/trust-model.md`.
- **Known gap (demo only):** The demo server's keypair generation/import endpoints are explicitly flagged as requiring authentication before production use.

## btc-yield

- **Structured audit program:** Five sequential third-party audits (Mar–Apr 2026) with tracked Critical-to-Medium findings and corresponding fix commits, covering on-chain validation gaps, credential exposure, and API authorization failures.
- **On-chain validation:** The indexer's transaction processor enforces canonical outpoint spending for provider-claims (H-5-01), withdrawal destination/amount binding (H-5-02), strict 1:1 output-to-allocation mapping with an AML invariant (C-5-01), and explicit deposit outpoint validation to block attacker-controlled outputs (H-5-03).
- **API input validation:** NestJS global `ValidationPipe` is configured fail-closed (`forbidNonWhitelisted: true`). DTOs enforce UUID formats, numeric bounds, and PSBT semantic validation. Configs reject missing required env vars at startup rather than falling back to unsafe defaults (M-5-04).
- **Secret management:** Runtime secrets injected via AWS Secrets Manager (replacing insecure Docker `ARG` patterns). CI uses AWS OIDC federation to eliminate long-lived credentials. Runtime env scripts guard against leaking sensitive variable names at output.
- **Static analysis pipeline:** GitHub Actions runs Semgrep, Gitleaks, CodeQL (fails at security-severity ≥ 4.0), Trivy, NPM Audit, and OSV on every push — all also runnable locally.
- **Infrastructure hardening:** Terraform provisions VPC public/private subnet isolation, security groups scoping inbound access to ALB/RDS/Bitcoin node, TLS via ACM/ALB HTTPS listeners, and RDS encryption-at-rest with credentials from Secrets Manager.
